# chezmoi Integration

How chezmoi-recipes integrates with [chezmoi](https://www.chezmoi.io/). For chezmoi's own features (source format, templates, scripts, externals, password managers), see the [chezmoi docs](https://www.chezmoi.io/reference/).

## Integration approach

chezmoi-recipes integrates via chezmoi's [`read-source-state.pre` hook](https://www.chezmoi.io/reference/configuration-file/hooks/). Before chezmoi reads its source state, the hook runs `chezmoi-recipes overlay`, which copies recipe files into the chezmoi source directory. chezmoi then reads the populated source directory and applies as normal.

The user's primary workflow is just `chezmoi apply`. chezmoi-recipes is invisible infrastructure.

The hook fires for any chezmoi command that reads source state (`apply`, `diff`, `status`, `data`, etc.), so the source directory is always up to date.

## Ownership model

chezmoi-recipes owns the chezmoi source directory (`~/.local/share/chezmoi-recipes/source/` by default, separate from chezmoi's default source dir). chezmoi owns `~/`. Data flows one direction:

```
recipe chezmoi/  -->  chezmoi source dir  -->  chezmoi apply  -->  ~/
```

Neither tool reaches into the other's domain.

## Apply flow

```
User runs: chezmoi apply

  1. read-source-state.pre hook fires
     chezmoi-recipes overlay:
       a. Overlay each recipe's chezmoi/ into source dir (alphabetically)
       b. Record state to ~/.local/share/chezmoi-recipes/

  2. chezmoi reads source state
  3. chezmoi applies (files, scripts, templates) to ~/
```

The overlay is idempotent: running it multiple times with unchanged recipes produces the same result.

## State tracking

chezmoi-recipes and chezmoi each maintain independent state:

| | chezmoi-recipes | chezmoi |
|---|---|---|
| **What** | Which recipes own which source files | File hashes, script execution history |
| **Where** | `~/.local/share/chezmoi-recipes/state.json` | chezmoi's internal database |
| **Answers** | "Which recipe placed this source file?" | "Is this target file up to date?" |

Neither tool reads the other's state.

## Script ordering

After overlay, chezmoi sees a flat `.chezmoiscripts/` directory and runs scripts in lexicographic order by filename. It has no concept of recipes.

Scripts should be self-contained: if a script depends on a tool being present, it should check for it and install it if missing.

When ordering genuinely matters (e.g., mise must be installed before anything that uses `mise install`), use numeric prefixes:

```
.chezmoiscripts/
  run_once_00_install-mise.sh
  run_once_01_install-neovim.sh
```

This is a convention, not a mechanism. chezmoi-recipes does not enforce script ordering.

## Conflict handling

Two recipes targeting the same chezmoi source path is an error. chezmoi-recipes detects this by checking its state store before overlaying. If recipe A already placed `dot_gitconfig` and recipe B also contains `dot_gitconfig`, the overlay fails with a message identifying both recipes and the conflicting path.

No silent overwrites.

## Remove vs undo

`chezmoi-recipes remove` deletes a recipe's files from the chezmoi source directory and removes its entry from state. It does not reverse side effects from chezmoi scripts (installed packages, system config, etc.). Tracking and reversing arbitrary script actions is out of scope.

## Per-recipe .chezmoiignore

chezmoi reads a single `.chezmoiignore` from the source directory. Since recipes are independent, each recipe can include its own `chezmoi/.chezmoiignore` with ignore entries (typically wrapped in template conditionals). During a full overlay (no named arguments), chezmoi-recipes collects all per-recipe `.chezmoiignore` files and merges them into the single `.chezmoiignore` in the source directory.

The merged file is fully regenerated on every overlay run. No section markers, no partial updates. The format:

```
# Generated by chezmoi-recipes. DO NOT EDIT.

# Global
scripts/

# Recipe: alacritty
{{ if .isContainer }}
private_dot_config/alacritty/
{{ end }}

# Recipe: cartage
{{ if .isContainer }}
.chezmoiscripts/enable-cartage.sh
private_dot_config/systemd/user/cartage.service
{{ end }}
```

Recipe entries can contain Go template syntax. chezmoi-recipes passes them through verbatim; chezmoi evaluates templates at apply time.

Key behaviors:

- **Full overlays only.** Named overlays (`chezmoi-recipes overlay git`) do not regenerate `.chezmoiignore`, since that would lose other recipes' entries.
- **`scripts/` always present.** The global `scripts/` entry is always included (prevents chezmoi from deploying shared script utilities to `~/`).
- **Not tracked in state.** The `.chezmoiignore` file is not recorded in the recipe state store. It is a derived artifact, not a recipe-owned file.
- **Idempotent.** Running overlay twice with unchanged recipes produces the same `.chezmoiignore` content.

For filtering entire recipes by environment (instead of individual files), use `.recipeignore`. Per-recipe `.chezmoiignore` is for finer-grained control within a recipe.

## Open questions

- **`.chezmoitemplates/` aggregation.** Template partials placed in recipe `chezmoi/.chezmoitemplates/` directories are handled naturally by overlay (each file gets a unique name, no merge needed). No special handling required.
- **Bootstrap on fresh systems.** An `install.sh` at the repo root is the recommended pattern: install chezmoi and chezmoi-recipes to `~/.local/bin`, run `chezmoi-recipes init --recipes-dir <repo>/recipes`, then `chezmoi init --source ~/.local/share/chezmoi-recipes/source` to prompt for user data, then `chezmoi apply`. See [fgrehm/home](https://github.com/fgrehm/home) for a reference implementation.
