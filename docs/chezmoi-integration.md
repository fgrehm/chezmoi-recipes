# chezmoi Integration

How chezmoi-recipes integrates with [chezmoi](https://www.chezmoi.io/). For chezmoi's own features (source format, templates, scripts, externals, password managers), see the [chezmoi docs](https://www.chezmoi.io/reference/).

## Architecture

chezmoi-recipes uses chezmoi's [`.chezmoiroot`](https://www.chezmoi.io/reference/special-files/chezmoiroot/) feature. The user's dotfiles repo is the chezmoi working tree. A `.chezmoiroot` file at the repo root points chezmoi at a gitignored `compiled-home/` directory, which is rebuilt from scratch on every `chezmoi apply`.

```
~/dotfiles/                     <- chezmoi working tree (git repo root)
  .chezmoiroot                  <- contains "compiled-home" (tracked)
  .gitignore                    <- includes compiled-home/
  home/                         <- freeform chezmoi source files (tracked)
    .chezmoi.toml.tmpl
    dot_bashrc
  recipes/                      <- recipe directories (tracked)
    git/
      README.md
      chezmoi/
        dot_gitconfig
  compiled-home/                <- gitignored, generated by overlay
    .chezmoi.toml.tmpl
    dot_bashrc
    dot_gitconfig
```

`home/` contains chezmoi source files that don't belong to any recipe (config template, shell configs, etc.). `recipes/` contains recipe directories. `compiled-home/` is the merge of both, rebuilt on every overlay run.

## Integration hook

chezmoi-recipes integrates via one [chezmoi hook](https://www.chezmoi.io/reference/configuration-file/hooks/):

- **`read-source-state.pre`** runs `chezmoi-recipes overlay`, which clears `compiled-home/`, copies `home/` files, then overlays each recipe's `chezmoi/` directory. This fires for any chezmoi command that reads source state (`apply`, `diff`, `status`, `data`, etc.), so `compiled-home/` is always up to date.

The user's primary workflow is just `chezmoi apply`. chezmoi-recipes is invisible infrastructure.

## Apply flow

```
User runs: chezmoi apply

  1. read-source-state.pre hook fires
     chezmoi-recipes overlay:
       a. Clear compiled-home/
       b. Copy home/ -> compiled-home/
       c. Deploy shared scripts (scripts/ui.bash)
       d. Overlay each recipe's chezmoi/ into compiled-home/ (alphabetically)
       e. Merge per-recipe .chezmoiignore
       f. Save state

  2. chezmoi reads source state from compiled-home/ (via .chezmoiroot)
  3. chezmoi applies (files, scripts, templates) to ~/
```

The overlay is idempotent: running it multiple times with unchanged recipes produces the same result. Since `compiled-home/` is cleared and rebuilt from scratch, there are no stale files.

## `chezmoi update` works natively

Because the dotfiles repo is the chezmoi working tree and has a git remote, `chezmoi update` works as expected:

1. chezmoi runs `git pull` on the working tree (the dotfiles repo)
2. `read-source-state.pre` hook fires, rebuilding `compiled-home/`
3. chezmoi applies the updated source state

No workarounds needed.

## Guard hooks

The config template includes `.pre` hooks that block commands which would write to `compiled-home/`:

- `chezmoi add`, `chezmoi edit`, `chezmoi re-add`
- `chezmoi merge`, `chezmoi chattr`, `chezmoi import`
- `chezmoi forget`, `chezmoi destroy`

These commands don't make sense with a generated source directory. The guard hooks print an error message and exit non-zero, preventing chezmoi from proceeding. Users should edit files in `home/` or `recipes/` directly instead.

Commands that only read source state (`apply`, `diff`, `status`, `managed`, `doctor`, `data`) are not guarded.

## Ownership model

`home/` and `recipes/` are the source of truth (tracked in git). `compiled-home/` is a derived artifact (gitignored). chezmoi reads `compiled-home/` and owns `~/`. Data flows one direction:

```
home/ + recipes/  -->  compiled-home/  -->  chezmoi apply  -->  ~/
```

A file must belong to either `home/` or exactly one recipe. If the same file exists in both `home/` and a recipe's `chezmoi/` directory, the overlay fails with a conflict error.

## State tracking

chezmoi-recipes and chezmoi each maintain independent state:

| | chezmoi-recipes | chezmoi |
|---|---|---|
| **What** | Which recipes own which source files | File hashes, script execution history |
| **Where** | `~/.local/share/chezmoi-recipes/state.json` | chezmoi's internal database |
| **Answers** | "Which recipe placed this source file?" | "Is this target file up to date?" |

State is reset on each overlay run since `compiled-home/` is rebuilt from scratch.

## Script ordering

After overlay, chezmoi sees a flat `.chezmoiscripts/` directory and runs scripts in lexicographic order by filename. It has no concept of recipes.

Scripts should be self-contained: if a script depends on a tool being present, it should check for it and install it if missing.

When ordering genuinely matters (e.g., mise must be installed before anything that uses `mise install`), use numeric prefixes:

```
.chezmoiscripts/
  run_once_00_install-mise.sh
  run_once_01_install-neovim.sh
```

This is a convention, not a mechanism. chezmoi-recipes does not enforce script ordering.

## Conflict handling

Two types of conflicts are detected:

1. **Home/recipe conflict.** A file exists in both `home/` and a recipe's `chezmoi/` directory. Detected before any copying. Each file must belong to either `home/` or exactly one recipe.

2. **Recipe/recipe conflict.** Two recipes contain the same chezmoi source path. Detected during the overlay loop via the in-memory state store.

Both produce clear error messages identifying the conflicting paths and sources. No silent overwrites.

## Remove vs undo

`chezmoi-recipes remove` deletes a recipe's files from `compiled-home/` and removes its entry from state. It does not reverse side effects from chezmoi scripts (installed packages, system config, etc.). The next overlay run will re-overlay the recipe unless the recipe directory is deleted or added to `.recipeignore`.

## Per-recipe .chezmoiignore

chezmoi reads a single `.chezmoiignore` from the source directory. Since recipes are independent, each recipe can include its own `chezmoi/.chezmoiignore` with ignore entries (typically wrapped in template conditionals). During a full overlay (no named arguments), chezmoi-recipes collects all per-recipe `.chezmoiignore` files and merges them into the single `.chezmoiignore` in `compiled-home/`.

The merged file is fully regenerated on every overlay run. The format:

```
# Generated by chezmoi-recipes. DO NOT EDIT.

# Global
scripts/

# Recipe: alacritty
{{ if .isContainer }}
private_dot_config/alacritty/
{{ end }}

# Recipe: cartage
{{ if .isContainer }}
.chezmoiscripts/enable-cartage.sh
private_dot_config/systemd/user/cartage.service
{{ end }}
```

Recipe entries can contain Go template syntax. chezmoi-recipes passes them through verbatim; chezmoi evaluates templates at apply time.

Key behaviors:

- **Full overlays only.** Named overlays (`chezmoi-recipes overlay git`) do not regenerate `.chezmoiignore`, since that would lose other recipes' entries.
- **`scripts/` always present.** The global `scripts/` entry is always included (prevents chezmoi from deploying shared script utilities to `~/`).
- **Not tracked in state.** The `.chezmoiignore` file is not recorded in the recipe state store. It is a derived artifact, not a recipe-owned file.
- **Idempotent.** Running overlay twice with unchanged recipes produces the same `.chezmoiignore` content.

For filtering entire recipes by environment (instead of individual files), use `.recipeignore`. Per-recipe `.chezmoiignore` is for finer-grained control within a recipe.

## Bootstrap flow

On a fresh system:

```bash
# 1. Install chezmoi and chezmoi-recipes
sh -c "$(curl -fsSL .../install.sh)" -- username

# Or manually:
# git clone <repo> ~/dotfiles
# chezmoi-recipes init --recipes-dir ~/dotfiles/recipes
# chezmoi init --source ~/dotfiles
# chezmoi apply
```

The install script clones the dotfiles repo, runs `chezmoi-recipes init` (creates `.chezmoiroot`, `home/`, config template, initial overlay), then `chezmoi init --source ~/dotfiles` (processes the config template, prompts for user data), then `chezmoi apply`.
